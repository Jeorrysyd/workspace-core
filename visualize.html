服务群聊 — 我们是怎么协作的

工作台服务群

8 位成员
App
app.js — 总控
大家好，我是工作台的总调度。所有房间通过 app.register() 向我注册，我管导航切换和状态栏。最重要的是我维护了一个 事件总线，房间之间通过 app.emit() 和 app.on() 互传数据。比如档案馆点"发送到写作"，我就 emit 一个 writing:receive 事件。
档案
档案馆 — 内容源头
我是整个工作台的数据基座。我从两个地方拿数据：一是 ~/ObsidianVault/voice-context/（voice-to-context 项目每天生成的日志、灵感、消费记录），二是用户直接在工作台上传的录音。每条档案都有"发送到写作"和"深度探索"按钮，内容直接通过事件总线流转。
W
Whisper CLI — 转写引擎
档案馆上传录音时会叫我。流程是 ffmpeg 转 16kHz WAV → 我用本地大模型转写 → 文本交给 Claude 分类。我跑在本地，不走网络 API，574MB 的 large-v3-turbo 模型就在 ~/tools/ 里。处理一段 5 分钟录音大概要 30 秒。
AI
Claude 服务 — AI 中枢
我是所有 AI 能力的统一出口。server/services/claude.js 封装了三种调用方式：单轮流式（写作用）、多轮对话流式（对话用）、同步完成（档案分类用）。全部走 SSE 流式传输，用户看到的是字一个一个冒出来的效果。
Mem
记忆服务 — 个人上下文
我维护一个 data/memory/memory.json，存 用户 的个人上下文碎片。每次档案馆处理完录音就往我这追加一条。写作房间和对话房间的 AI 调用都会先找我拿最近 20 条记忆作为 prompt 背景，这样 AI 就"认识"用户 了。
写
写作房间 — 内容产出
我是内容生产的终点站。我有四个 AI 模式：选题灵感、口播稿、内容优化、对话式写作。每个模式的 prompt 都注入了 用户 的个人记忆和最近档案，所以 AI 生成的选题不是泛泛而谈，而是真正来自用户的经历。我还能接收来自档案馆、播客、对话的内容，直接变成草稿。
播
播客模块 — 外部知识
我代理了 ~/Desktop/podcast-analyze/ 项目的数据。那个项目用 Deepgram 转录 + Claude 分析，我直接读它的 JSON 结果。我有完整的音频播放器：可拖拽进度条、15秒快进快退、时间线自动同步、章节导航。点击任何时间戳都能跳转播放。还能把播客内容一键发到写作或对话探索。
话
对话模块 — 自我探索
我是工作台里最"有温度"的部分。我的 AI prompt 把 用户 定位为对话伙伴而不是用户。我读取全量个人上下文（记忆 + 档案 + Obsidian日志 + 灵感），然后用四种起始模式引导探索：复盘今天、思维盲点、连接灵感、挑战假设。其他房间也能把内容发到我这里做深度探索。
App
app.js — 总控
总结一下我们的协作关系：

数据流向：录音→档案馆→记忆→AI上下文→写作/对话
跨房间通信：事件总线 emit/on 模式，任何房间都能发送内容到写作或对话
AI 上下文：写作和对话都读取 Memory + Obsidian + Archive 构建完整的 用户 画像
外部项目集成：Obsidian (只读导入) + podcast-analyze (数据代理) + Whisper CLI (本地转写)
写
写作房间
补充一下：关于"多路 agent 并行"的问题 — 目前我们虽然跑在一个 Node.js 进程里，但架构上已经是 模块化 agent 模式。每个房间就是一个 agent（有自己的专属 system prompt 和工具），通过事件总线协作。下一步如果要做真正的并行，可以把每个 agent 拆成独立进程/Worker，通信方式从内存事件改为消息队列，架构不用大改。
